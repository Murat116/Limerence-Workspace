# Plain Language Style — артефакты Feature Delivery

Все файлы в `.cursor/feature-work/<slug>/` должны читаться **без погружения в код и без контекста чата**.

## Обязательная структура каждого артефакта

1. **Резюме** (3–5 предложений) — о чём задача, сколько проблем, что решить первым
2. **Основной текст** — простой русский, без жаргона
3. **Глоссарий** — любой термин из кода/БД/Figma с переводом
4. **Детали для агента** — `<details>` или подсекция; пути, SQL, enum — только там

## Правила формулировок

| ❌ Не так | ✅ Так |
|----------|--------|
| `EntitlementResolver.canAccess` блокирует | Без подписки или пропуска пользователь не может открыть главу |
| `scenario_id: offline_read_blocked` | Экран «нужен интернет / подписка» при чтении офлайн без Pass |
| «см. PaywallPresenter» | «Окно покупки показывается через …» + путь в деталях |
| Пустая ячейка в источнике | «не описано» |
| `PW-01`, `bn01`, `T1` | Человекочитаемое имя сценария из spec |

## Имена источников в тексте

Используй **понятные подписи**, не внутренние коды:

- **Приложение (как сейчас работает)** — RN code
- **Спека в репо** — `Спецификация/`
- **Конструктор для авторов** — LimerenceBilder
- **База данных** — Supabase / migrations
- **Макет** — Figma
- **Задача в Notion** — Notion page

## Цитаты из первоисточников

Формат:

```markdown
- **Спека в репо:** 7 дней ожидания — `Спецификация/.../Глава.md`, строки 42–44:
  > «Подписчик получает ранний доступ за N дней до релиза»
```

## Severity в сводках

| Emoji | Уровень | Подпись для человека |
|-------|---------|----------------------|
| 🔴 | blocker | Блокирует реализацию — нужно решение |
| 🟠 | major | Важно согласовать до кода |
| 🟡 | minor | Можно отложить, но зафиксировать |
| ⚪ | info | Для сведения, не блокирует |

## Блок «Что это значит для задачи»

Одно предложение — практическое следствие, не пересказ фактов.

**Плохо:** «В коде 3, в spec 7.»  
**Хорошо:** «Пока не согласуем число дней, нельзя править UI главы и валидацию в конструкторе.»
