# Техническая спецификация проекта

Этот документ описывает ключевые технические и архитектурные решения, принятые в проекте.

---

## 1. Архитектура сервисного слоя

Приложение использует многоуровневую архитектуру для разделения ответственности и упрощения поддержки.

- **`StoryService.ts`**:
  - **Роль**: Фасад (Facade) для всего UI-слоя.
  - **Описание**: Является единственной точкой входа для компонентов и Redux Thunks. Он инкапсулирует логику взаимодействия с кешем (`StoryManager`) и сетью (`StoryRepository`), предоставляя простые методы (`getChapterDetails`, `savePlayerStatsForStory` и т.д.). UI-слой не знает, откуда приходят данные — из кеша или из сети.

- **`StoryManager.ts`**:
  - **Роль**: Менеджер кеша и локального хранилища.
  - **Описание**: Взаимодействует напрямую с `AsyncStorage`. Отвечает за две основные задачи:
    1.  **Кеширование контента**: Сохраняет DTO (Data Transfer Objects), полученные из сети (главы, персонажи, статы), чтобы минимизировать повторные загрузки.
    2.  **Сохранение прогресса игрока**: Записывает и считывает данные, специфичные для пользователя, такие как его характеристики (`player_stats`) и сделанные выборы (`completedVariantIds`).

- **`StoryRepository.ts`**:
  - **Роль**: Слой доступа к данным (Data Access Layer).
  - **Описание**: Отвечает исключительно за сетевые запросы (например, к Supabase или другому API). Возвращает "сырые" DTO, которые затем обрабатываются `StoryManager` и `StoryService`.

## 2. Управление состоянием (Redux Toolkit)

Глобальное состояние приложения управляется с помощью Redux Toolkit, что обеспечивает предсказуемость и удобную отладку.

- **Slices**: Каждый доменный модуль (`quest`, `home`, `chapterEnding`, etc.) имеет свой "слайс", который содержит:
  - `initialState`: Начальное состояние модуля.
  - `reducers`: Синхронные функции для изменения состояния.
  - `extraReducers`: Обработчики асинхронных действий (thunks), которые обновляют состояние по результатам их выполнения (pending, fulfilled, rejected).

- **Thunks**:
  - **Роль**: Обработка асинхронной логики и побочных эффектов.
  - **Описание**: Используются для выполнения сложных операций, таких как загрузка данных из `StoryService`, их обработка и последующий диспатч синхронных actions для обновления состояния. Пример: `initializeQuest` в `quest/thunks.ts`.

## 3. Поток данных и персистентность

### 3.1. Запуск квеста

1.  **UI (`ChapterStartScreen`)**: Пользователь нажимает "Начать главу".
2.  **Dispatch**: Вызывается `initializeQuest({ chapterId, storyId })`.
3.  **Thunk (`initializeQuest`)**:
    -   Обращается к `StoryService.getChapterDetails(chapterId)` для загрузки данных главы.
    -   Обращается к `StoryService.getPlayerStatsForStory(storyId)` для загрузки прогресса игрока.
    -   `StoryService` делегирует эти вызовы `StoryManager`.
    -   `StoryManager` пытается прочитать данные из `AsyncStorage`. Если их там нет, `StoryService` (в будущем) обратится к `StoryRepository`.
4.  **Reducer**: Thunk завершается (fulfilled), и редьюсер в `questSlice` обновляет состояние (`status`, `chapter`, `scene`, `stats` и т.д.).
5.  **UI (`SceneScreen`)**: Реагирует на обновление состояния и отображает первую сцену.

### 3.2. Выбор варианта (сохранение прогресса)

1.  **UI (`SceneScreen`)**: Пользователь нажимает на кнопку варианта.
2.  **Dispatch**: Вызывается thunk `handleSelectVariant(variant)`.
3.  **Thunk (`handleSelectVariant`)**:
    -   **Синхронно**: Немедленно диспатчит action `selectVariant(variant)`, чтобы UI мгновенно отреагировал.
    -   **Reducer (`selectVariant`)**: Обновляет `state.stats` и `state.completedVariantIds`.
    -   **Асинхронно**: Thunk получает обновленное состояние (`getState()`) и вызывает `StoryService.savePlayerStatsForStory(storyId, newStats)`.
    -   `StoryService` передает вызов в `StoryManager`, который записывает новые статы в `AsyncStorage`.
    -   **Диспатч `handleNextReplica()`**: После всех операций запускается переход к следующей реплике/сцене.

Эта архитектура обеспечивает **отказоустойчивость**: даже если пользователь закроет приложение в середине главы, его прогресс по статам будет сохранен.
