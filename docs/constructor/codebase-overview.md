# Документация по коду LimerenceBilder

Это подробная документация по кодовой базе проекта LimerenceBilder. 
Она описывает архитектуру, компоненты, хранилища состояния и утилиты, используемые в приложении.

## Обзор проекта

LimerenceBilder - это визуальный редактор для создания интерактивных историй. 
Он позволяет пользователям создавать разветвленные сюжеты с помощью графового интерфейса, управлять персонажами, статами и диалогами.

## Структура проекта

Основная логика приложения находится в директории `editor-app`. 
Это React-приложение, созданное с помощью Vite и использующее TypeScript.

- `editor-app/src`: Исходный код приложения.
- `editor-app/src/components`: Компоненты React, формирующие UI.
- `editor-app/src/store`: Хранилища состояния (Zustand).
- `editor-app/src/types`: Определения типов TypeScript.
- `editor-app/src/utils`: Вспомогательные функции.
- `editor-app/src/assets`: Статические ресурсы (изображения, иконки).

## Точка входа: `main.tsx`

Файл `editor-app/src/main.tsx` является точкой входа в приложение.

### Назначение

Этот файл отвечает за инициализацию и рендеринг корневого компонента React (`<App />`) в DOM.

### Код и описание

```typescript
import { StrictMode } from 'react'
import { createRoot } from 'react-dom/client'
import './index.css'
import App from './App.tsx'

createRoot(document.getElementById('root')!).render(
  <StrictMode>
```

- **`createRoot`**: Используется для создания "корня" React 18, который позволяет использовать конкурентные возможности React.

- **`<StrictMode>`**: Обертка, которая помогает выявлять потенциальные проблемы в приложении, активируя дополнительные проверки и предупреждения.
- **`<App />`**: Корневой компонент всего приложения.

## Корневой компонент: `App.tsx`

Файл `editor-app/src/App.tsx` определяет основную структуру пользовательского интерфейса.

### Назначение

Этот компонент создает главный макет приложения, который разделен на три основные части: дерево проекта, редактор графа и инспектор свойств.

### UI и Структура

Приложение использует библиотеку `react-resizable-panels` для создания изменяемых по размеру панелей.

- **Верхняя панель (`<TopBar />`)**: Находится вверху и содержит основные действия, такие как сохранение и загрузка проекта.
- **Основной контейнер**: Разделен на три вертикальные панели:
    1.  **Левая панель (`<ProjectTree />`)**: Отображает иерархическую структуру проекта (книга, главы, статы). Занимает по умолчанию 20% ширины.
    2.  **Центральная панель (`<FlowEditor />`)**: Главная рабочая область, где отображается и редактируется граф истории (сцены, диалоги). Занимает по умолчанию 55% ширины.
    3.  **Правая панель (`<Inspector />`)**: Отображает свойства и настройки для выбранного элемента (сцены, диалога, персонажа и т.д.). Занимает по умолчанию 25% ширины.

- **`useEffect`**: При первой загрузке приложения этот хук вызывает `setActiveChapterId`, чтобы принудительно обновить граф и отобразить начальную главу. Это гарантирует, что `FlowEditor` не будет пустым при запуске.

### Неиспользуемые компоненты в `App.tsx`

**Важное замечание**: В файле `App.tsx` определены несколько компонентов (`StoryEditor`, `ChapterEditor`, `StatsEditor`, `MainContent`), которые, судя по всему, **не используются** в итоговой сборке приложения. Основной интерфейс строится вокруг `ProjectTree`, `FlowEditor` и `Inspector`. Логика выбора и отображения редакторов для разных сущностей (книги, главы, статов) реализована внутри компонента `Inspector`, а не через `MainContent`.

Ниже приведено описание этих неиспользуемых компонентов для полноты документации.

#### `StoryEditor`

- **Назначение**: Форма для редактирования общих свойств истории (название, автор, описание, обложка).
- **UI**: Содержит текстовые поля для названия и автора, текстовую область для описания и кнопку для загрузки изображения обложки.
- **Логика**: Использует `useStoryStore` для получения и обновления данных о книге. `useRef` используется для программного вызова диалога выбора файла.

#### `ChapterEditor`

- **Назначение**: Форма для редактирования свойств выбранной главы (название, описание).
- **UI**: Содержит текстовые поля для названия и описания главы.
- **Логика**: Принимает `chapterId` в качестве пропа. Использует `useStoryStore` для получения данных о конкретной главе и их обновления. Содержит проверку на случай, если глава не найдена (например, была удалена).

#### `StatsEditor` и `StatItem`

- **Назна-ение**: Интерфейс для управления пользовательскими статами (например, харизма, сила).
- **UI**:
    - `StatsEditor`: Отображает заголовок, кнопку "Добавить стат" и список всех статов.
    - `StatItem`: Отдельный элемент списка для одного стата. Содержит выпадающий список для выбора иконки, текстовое поле для названия стата и кнопку для его удаления.
- **Логика**:
    - `StatsEditor` получает все `statIds` из `useStoryStore` и рендерит компонент `StatItem` для каждого из них. Кнопка "Добавить стат" вызывает соответствующий экшен в `storyStore`.
    - `StatItem` является "гранулярным" компонентом, то есть он подписывается только на данные того стата, который он отображает. Это оптимизация, предотвращающая лишние перерисовки всего списка при изменении одного элемента.

#### `MainContent`

- **Назначение**: Компонент-маршрутизатор, который должен был отображать один из редакторов (`StoryEditor`, `ChapterEditor`, `StatsEditor`) или `FlowEditor` в зависимости от состояния, хранящегося в `useSelectionStore`.
- **Логика**: Использует `viewMode` и `viewModePayload` из `selectionStore` для условного рендеринга.
- **Статус**: Не используется. Вместо него, центральная панель всегда показывает `FlowEditor`, а правая панель (`Inspector`) берет на себя роль отображения контекстно-зависимых редакторов.

### `ProjectTree.tsx`

Компонент для левой панели, отображающий иерархическую структуру всего проекта.

#### Назначение и UI

Дерево проекта предоставляет обзор всех сущностей истории и позволяет пользователю перемещаться между ними. Оно разделено на секции:

- **Книга**: Корневой элемент, клик по которому открывает модальное окно `BookModal` для редактирования настроек истории.
- **Главы**: Список глав. У каждой главы есть кнопка `+` для добавления новой главы. Клик по главе делает ее активной (загружая ее граф в `FlowEditor`) и открывает `ChapterInspector`. Если глава активна, под ней отображается список ее сцен.
    - **Сцены**: Список сцен внутри активной главы. Клик по сцене выделяет ее в `FlowEditor` и открывает `SceneInspector`.
- **Персонажи**: Список всех персонажей. Клик по персонажу открывает `CharacterInspector`.
- **Статы**: Список всех пользовательских статов. Клик по стату открывает `StatInspector`.

Для каждой секции (`Главы`, `Персонажи`, `Статы`) есть кнопка `+` для быстрого добавления нового элемента.

#### Логика

- **`useStoryStore`**: Используется для получения всего дерева проекта (`project`) и для вызова экшенов добавления новых элементов (`addChapter`, `addCharacter`, `addStat`).
- **`useSelectionStore`**: Используется для установки текущего выделенного элемента (`select`). Это состояние затем используется `Inspector`-ом для отображения нужного редактора.
- **`useReactFlow`**: Используется для взаимодействия с `FlowEditor`. В частности, функция `setCenter` позволяет сфокусироваться на узле сцены и приблизить камеру при клике на сцену в дереве.
- **`handleSelect`**: Центральная функция для обработки кликов. Она вызывает `select` из `selectionStore` и, если выбранный элемент - это сцена, вызывает `handleNodeFocus`.
- **`handleChapterClick`**: Особый обработчик для глав, который не только выделяет главу, но и делает ее активной с помощью `setActiveChapterId`, что приводит к перерисовке графа в `FlowEditor`.

### `FlowEditor.tsx`

Это центральный и самый сложный компонент приложения, представляющий собой редактор графа на базе `react-flow`.

#### Назначение и UI

`FlowEditor` — это холст, на котором пользователи могут визуально конструировать логику истории.

- **Узлы (Nodes)**: Представляют собой элементы истории, такие как Сцены, Списки диалогов и отдельные Реплики. Для каждого типа узла есть свой кастомный компонент (например, `SceneNode`).
- **Рёбра (Edges)**: Соединяют узлы, показывая переходы между ними (например, от варианта ответа в диалоге к следующей реплике).
- **Кнопка `+`**: Расположена в правом верхнем углу и позволяет добавить новую сцену в текущую активную главу.
- **Панель инструментов (`FlowToolbar`)**: Плавающая панель внизу по центру, позволяющая переключать режимы взаимодействия с холстом:
    - **Рука (`pan`)**: Для перемещения по холсту.
    - **Курсор (`cursor`)**: Для выделения и перетаскивания узлов.
    - **Выделение (`selection`)**: Для выделения группы узлов рамкой.
- **Индикатор масштаба (`ZoomIndicator`)**: Небольшой плавающий элемент в правом нижнем углу, который показывает текущий уровень зума и позволяет его менять.

- **`useFlowStore`**: `FlowEditor` получает все данные для отображения (`nodes`, `edges`) и стандартные обработчики (`onNodesChange`, `onEdgesChange`, `onConnect`) из `flowStore`. Это позволяет отделить логику управления состоянием графа от его отображения.
- **`useStoryStore`**: Используется для вызова экшена `addScene` при нажатии на кнопку `+`.
- **`useSelectionStore`**: Используется для установки выделенного узла (`select`) или для сброса выделения (`clearSelection`) при клике на пустую область холста.
- **`nodeTypes` и `edgeTypes`**: Здесь происходит регистрация кастомных React-компонентов, которые будут использоваться для рендеринга узлов и рёбер вместо стандартных.
- **`handleAddScene`**: При добавлении новой сцены, ее позиция рассчитывается так, чтобы она появилась в центре видимой области холста. Для этого используется `reactFlowInstance.screenToFlowPosition`.
- **Пропсы `ReactFlow`**: Множество пропсов `ReactFlow` привязаны к состоянию из `flowStore` и `selectionStore`, что делает компонент полностью управляемым извне. Например, `nodesDraggable` зависит от `toolMode`, позволяя включать и выключать перетаскивание узлов.

### `Inspector.tsx` (и дочерние инспекторы)

Это компонент для правой панели, который динамически отображает релевантный набор полей для редактирования в зависимости от того, какой элемент выбран пользователем в `ProjectTree` или `FlowEditor`.

#### Назначение и Логика

`Inspector` работает как маршрутизатор. Он получает `selectedId` и `selectedType` из `useSelectionStore`. На основе `selectedType` он решает, какой из дочерних инспекторов нужно отобразить, и передает ему `selectedId`.

Если ничего не выбрано, инспектор отображает приветственное сообщение.

```typescript
// В компоненте Inspector
function Inspector() {
  const { selectedId, selectedType } = useSelectionStore();

  const getTitle = () => { /* ... возвращает заголовок ... */ };

  const renderInspector = () => {
    if (!selectedId || !selectedType) {
      return <p>Ничего не выбрано</p>;
    }
    // Рендерит нужный инспектор в зависимости от selectedType
    switch (selectedType) {
      case 'scene': return <SceneInspector sceneId={selectedId} />;
      case 'dialogList': return <DialogListInspector listId={selectedId} />;
      case 'dialog': return <DialogInspector dialogId={selectedId} />;
      case 'character': return <CharacterInspector charId={selectedId} />;
      case 'stat': return <StatInspector statId={selectedId} />;
      // Тип 'book' больше не используется - настройки истории открываются в модальном окне
      case 'chapter': return <ChapterInspector chapterId={selectedId} />;
      default: return <p>Инспектор не найден</p>;
    }
  };

  return (
    <div className="bg-white h-full p-4 overflow-y-auto">
      <h3 className="text-lg font-bold mb-4">{getTitle()}</h3>
      {renderInspector()}
    </div>
  );
}
```

Ниже описаны все дочерние компоненты-инспекторы.

#### `BookModal.tsx`

- **UI**: Модальное окно с полями "Название книги", "Автор", "Описание" и управлением рамками для реплик.
- **Логика**: Получает данные о книге из `useStoryStore` и обновляет их с помощью `setBookDetails`. Открывается при клике на название истории в дереве проекта.

#### `ChapterInspector.tsx`

- **UI**: Форма с полями "Название главы", "Описание главы".
- **Логика**: Принимает `chapterId`. Получает данные о главе из `useStoryStore` и обновляет их с помощью `updateChapter`.

#### `StatInspector.tsx`

- **UI**: Форма для редактирования одного стата. Включает поле для названия, большую кнопку для выбора иконки и кнопку "Удалить стат".
- **Логика**:
    - Принимает `statId`. Использует `useStoryStore` для получения, обновления (`updateStat`) и удаления (`deleteStat`) данных.
    - **`IconPicker`**: При клике на кнопку с иконкой открывается модальное окно `IconPicker`, которое отображает большой список иконок, сгруппированных по категориям из `data/statIcons.ts`. Выбор иконки закрывает окно и обновляет стат.

#### `SceneInspector.tsx`

Это один из самых комплексных инспекторов, отвечающий за настройку сцены.

- **UI**: Содержит множество секций:
    - **Название**: Текстовое поле для названия сцены.
    - **Фоновое изображение**: Кнопка для загрузки фона. После загрузки отображается превью и кнопка для удаления фона.
    - **Аудиодорожки**: Секция для управления фоновой музыкой или звуками. Можно добавлять несколько треков, загружать для них аудиофайлы, регулировать громкость, включать зацикливание и удалять.
    - **Минимально требуемые статы**: Секция, где можно задать условия на статы, которые должны быть у игрока для доступа к этой сцене. Например, "Сила > 10".
    - **Требуемые варианты ответов**: Секция, где можно указать, какие выборы в предыдущих диалогах должен был сделать игрок, чтобы попасть в эту сцену.
    - **Кнопка "Удалить сцену"**.
- **Логика**:
    - Использует `useStoryStore` для чтения и обновления всех данных сцены.
    - Для загрузки файлов (аудио, фон) используется `URL.createObjectURL` для создания временной локальной ссылки на файл.
    - Сложные структуры, такие как `minimumRequiredStats`, обновляются путем создания копии объекта, изменения и последующей передачи в `updateScene`.

#### `DialogListInspector.tsx`

Инспектор для узла "Список диалогов" - группы связанных реплик.

- **UI**: Позволяет редактировать название списка, добавлять новые реплики (диалоги) в список, а также задавать условия видимости для всего списка (требуемые статы и варианты).
- **Логика**: Аналогична `SceneInspector` в части управления условиями. При добавлении нового диалога вызывает экшен `addDialogToScene`.

#### `DialogInspector.tsx`

Инспектор для узла "Диалог" - одной реплики персонажа.

- **UI**:
    - **Персонаж**: Выпадающий список для выбора говорящего персонажа.
    - **Текст реплики**: Текстовая область для самого диалога.
    - **Аудио**: Возможность прикрепить аудиофайл, который проиграется вместе с репликой.
    - **Эффекты камеры**: Поля для указания эффектов (например, тряска экрана).
    - **Варианты ответа**: Секция для добавления/редактирования/удаления вариантов, которые будут показаны игроку после этой реплики.
        - **Текст варианта**: Что увидит игрок.
        - **Условия показа**: Условия на статы, при которых вариант будет доступен.
        - **Изменения статов**: Какие статы и как изменятся, если игрок выберет этот вариант (например, `+1` к Харизме).
- **Логика**: Вся логика сводится к вызову `updateDialog` из `useStoryStore` с обновленными данными. Работа с вложенными структурами (варианты, изменения статов) требует аккуратного копирования и изменения массивов/объектов.

#### `CharacterInspector.tsx`

- **UI**: Простая форма для редактирования персонажа: поле для имени и чекбокс "Главный персонаж".
- **Логика**: Использует `updateCharacter` для изменения данных и `deleteCharacter` для удаления.

## Элементы графа (Nodes и Edges)

`FlowEditor` использует кастомные компоненты для отображения узлов и связей графа. Они находятся в `editor-app/src/components/nodes` и `editor-app/src/components/edges`.

### `SceneNode.tsx`

Это компонент для узла "Сцена". Сцены являются высокоуровневыми контейнерами для "Списков диалогов".

#### UI

- **Контейнер**: Узел сцены выглядит как большая область, в которую можно вкладывать другие узлы.
- **Авто-размер**: Сцена автоматически расширяется, чтобы вместить все дочерние узлы (`DialogListNode`), которые в ней находятся. Это достигается расчетом максимальных координат `x` и `y` дочерних узлов.
- **Фон**: Если в `SceneInspector` для сцены задано фоновое изображение, оно будет отображаться как полупрозрачный, размытый фон самого узла.
- **Заголовок**: Вверху узла отображается его название.
- **Кнопка `+`**: Позволяет добавить новый `DialogListNode` внутрь этой сцены.
- **Условия входа**: Если для сцены заданы условия (требуемые статы или варианты), они отображаются в небольшой плашке вверху узла.
- **Handles**: Точки для соединения связей. `target` (вход) слева, `source` (выход) справа.

#### Логика

- **`memo`**: Компонент обернут в `React.memo` для оптимизации, чтобы избежать лишних перерисовок, если его пропсы не изменились.
- **`useStoryStore`**: Используется для получения данных о сцене и для добавления новых списков диалогов (`addDialogList`).
- **Расчет размера**: Логика в `useMemo` перебирает все узлы из `flowStore`, находит дочерние (те, у которых `parentNode` равен `id` текущей сцены) и вычисляет минимально необходимые `width` и `height` для их вмещения.
- **Добавление списка диалогов**: Позиция нового списка диалогов рассчитывается так, чтобы он появился в центре видимой части узла-сцены.

### `DialogListNode.tsx`

Компонент для узла "Список диалогов", который является контейнером для отдельных реплик (`DialogNode`).

- **Состояние**:
  - Получает `id` (ID списка диалогов), `data` (`label`) и `selected` из пропсов `reactflow`.
  - Использует `useStoryStore` для доступа к данным списка (`dialogLists[id]`), всем диалогам, персонажам и сценам.
  - Использует `useSelectionStore` для обработки выбора дочерних `DialogNode`.
- **Функциональность**:
  - **Перетаскивание (Drag-and-Drop)**: Реплики внутри списка можно перетаскивать с помощью `dnd-kit` для изменения их порядка.
  - **Добавление реплик**: Кнопка `+` в заголовке добавляет новую реплику в список.
  - **Изменение размера**: Правую границу узла можно тянуть, чтобы изменить количество колонок для отображения реплик.
  - **Отображение требований**: Показывает условия (статы, выбранные варианты), необходимые для активации этого списка.
  - **Рендеринг дочерних узлов**: Отображает все реплики (`DialogNode`), принадлежащие этому списку, передавая им необходимые данные и статус `selected`.

### `Dialog/DialogNode.tsx`

Компонент для отображения узла "Диалог" (одна реплика).

- **Состояние**:
  - Получает `dialog`, `character`, и `selected` как пропсы.
- **Функциональность**:
  - Отображает текст реплики, имя персонажа и порядковый номер.
  - Позволяет редактировать текст реплики прямо в узле.
  - Подсвечивает рамку, если узел выбран (`selected={true}`).
  - Отображает варианты ответов, если они есть.

### `Dialog/DialogText.tsx`

Компонент для редактируемого текстового поля внутри `DialogNode`.

- **Состояние**:
  - Имеет внутреннее состояние `localText` для оптимизации (событие `onChange` не дергает глобальный стейт на каждое нажатие).
- **Функциональность**:
  - Автоматически подстраивает высоту под контент.
  - Обновляет глобальное состояние `onBlur` (когда фокус уходит с поля), если текст был изменен.

### `CustomEdge.tsx`

Компонент для кастомной связи (ребра) между узлами.

#### UI и Логика

Стандартное ребро в `react-flow` — это просто линия. `CustomEdge` расширяет его, добавляя метку прямо на линию связи.

- **`getBezierPath`**: Стандартная функция из `react-flow` для получения SVG-пути кривой Безье между двумя точками.
- **`EdgeLabelRenderer`**: Компонент из `react-flow`, который позволяет отобразить любой React-компонент (в данном случае `div` с текстом) поверх ребра.
- **Метка с условиями**:
    - `CustomEdge` получает `id` целевого узла (`target`).
    - Он использует `id` чтобы получить данные целевого узла из `storyStore`.
    - Если у целевого узла (обычно это сцена или список диалогов) есть условия входа (`minimumRequiredStats` или `requiredSelectedVariants`), они форматируются в строку.
    - Эта строка отображается как текст метки на ребре.
- **Результат**: Пользователь прямо в графе видит, какие условия нужно выполнить, чтобы переход по этой связи стал возможен. Например, на связи, ведущей к сцене "Бой с драконом", может появиться метка "Сила: 20".

## Управление состоянием (Zustand Stores)

Приложение использует `Zustand` для управления состоянием. Состояние разделено на несколько логических хранилищ (сторов).

### `storyStore.ts`

Это главный и самый важный стор. Он содержит все данные самого проекта: структуру книги, глав, сцен, диалогов, персонажей и т.д.

#### Структура состояния (`StoryState`)

- `project: Project`: Огромный объект, содержащий все данные истории. Его структура определена в `types/story.ts`. Он включает в себя словари (объекты) для `chapters`, `scenes`, `dialogLists`, `characters`, `userStats` и т.д., где ключами являются уникальные ID.
- `activeChapterId: ChapterId | null`: ID главы, которая в данный момент открыта в `FlowEditor`.

#### Ключевые экшены (Actions)

`storyStore` использует `immer` (через мидлвар `produce`) для безопасного и удобного изменения вложенных структур состояния.

- **`setProject(project)`**: Полностью заменяет текущий проект новым. Используется при импорте проекта. Также вызывает `buildNodesAndEdges` для перестройки графа в `flowStore`.
- **`setActiveChapterId(chapterId)`**: Устанавливает активную главу. Главный эффект этого экшена — полная перерисовка графа в `FlowEditor` на основе данных новой главы.
- **Экшены для создания (`add...`)**:
    - `addScene`, `addDialogList`, `addDialog`: Эти экшены не только добавляют новые данные в `project`, но и взаимодействуют с `flowStore`, вызывая `addNode` для добавления соответствующего узла на холст.
    - `addChapter`, `addCharacter`, `addStat`, `addVariant`: Добавляют новые сущности в `project`.
- **Экшены для обновления (`update...`)**:
    - `updateScene`, `updateDialogList`, `updateDialog` и т.д.: Обновляют данные конкретной сущности. Если изменяется свойство, влияющее на отображение в графе (например, `title`, который используется как `label` для узла), экшен также вызывает `updateNodeData` из `flowStore`.
- **Экшены для удаления (`delete...`)**:
    - `deleteSceneFromChapter`, `deleteDialogList`, `deleteDialog`: Удаляют сущность из `project`. Что важно, они также вызывают `deleteElements` из `flowStore`, чтобы удалить с холста сам узел и все связанные с ним ребра.
- **`syncNodePositions()`**: Вызывается после того, как пользователь закончил перетаскивать узел в `FlowEditor`. Этот экшен перебирает все узлы из `flowStore` и сохраняет их актуальные `position` в `project.nodePositions`. Это обеспечивает персистентность расположения узлов при сохранении/загрузке.

### `flowStore.ts`

Этот стор отвечает исключительно за состояние самого `ReactFlow` редактора.

#### Структура состояния (`FlowState`)

- `nodes: Node[]`: Массив узлов, который `ReactFlow` использует для рендеринга.
- `edges: Edge[]`: Массив ребер (связей).
- `toolMode: ToolMode`: Текущий выбранный инструмент на панели (`cursor`, `pan` или `selection`).

#### Ключевые экшены (Actions)

- **`onNodesChange`, `onEdgesChange`**: Стандартные обработчики `ReactFlow`. Они получают изменения (перетаскивание, выделение) от `ReactFlow` и применяют их к `nodes` и `edges` с помощью утилит `applyNodeChanges` и `applyEdgeChanges`.
- **`onConnect(connection)`**: Обработчик, который вызывается, когда пользователь пытается вручную соединить два узла. Содержит логику, запрещающую некорректные соединения (например, между узлами разных типов или в разных родительских контейнерах). Если соединение валидно, вызывает `addEdge`.
- **`addEdge(connection)`**: Добавляет новое ребро. Важно, что он устанавливает `type: 'custom'`, чтобы для рендера использовался наш `CustomEdge.tsx`.
- **`updateNodeData(nodeId, data)`**: Обновляет поле `data` у конкретного узла. Используется `storyStore` для синхронизации (например, при изменении названия сцены).
- **`setNodes(nodes)`, `setEdges(edges)`**: Полностью заменяют текущие узлы и ребра. Вызываются из `storyStore` при смене активной главы.
- **`deleteElements(...)`**: Удаляет узлы и ребра. Умная логика автоматически удаляет ребра, если удаляется один из узлов, которые они соединяют.

### `selectionStore.ts`

Самый простой стор, отвечающий за то, какой элемент в данный момент выбран в приложении.

#### Структура состояния (`SelectionState`)

- `selectedId: string | null`: ID выбранного элемента.
- `selectedType: SelectedItem | null`: Тип выбранного элемента (`scene`, `character` и т.д.).
- **`viewMode` и `viewModePayload`**: Поля, которые, вероятно, предназначались для неиспользуемой логики `MainContent` для переключения всего центрального вида. В текущей реализации не используются.

#### Экшены

- **`select(id, type)`**: Устанавливает выбранный элемент. Вызывается из `ProjectTree` и `FlowEditor`.
- **`clearSelection()`**: Сбрасывает выделение. Вызывается при клике на пустую область в `FlowEditor` или при удалении элемента.

### `initialProject.ts`

Это не стор, а просто файл, экспортирующий константу `initialProject`.

#### Назначение

- Определяет структуру и начальное состояние для нового, пустого проекта.
- Используется в `storyStore` как значение по умолчанию для `project`.
- Включает в себя двух "системных" персонажей: `char-main` (Главный герой) и `char-narrator` (Рассказчик), а также одну пустую главу.

## Вспомогательные модули

### `types/story.ts`

Этот файл является одним из самых важных в проекте, так как он определяет "схему данных" для всей истории. Он содержит определения TypeScript для всех сущностей проекта.

#### Ключевые типы

- **`type ...Id = string`**: В начале файла определяется множество псевдонимов (`type alias`) для ID разных сущностей (например, `SceneId`, `CharacterId`). Это улучшает читаемость кода, так как вместо `string` в сигнатурах функций можно видеть, ID какой именно сущности ожидается.
- **`Project`**: Корневой тип, который агрегирует все данные. Содержит `book`, а также словари (`Record<Id, Data>`) для `chapters`, `scenes`, `characters` и т.д.
- **Сущности (`Scene`, `Dialog`, `Character` и др.)**: Описывают структуру каждой отдельной сущности. Например, тип `Dialog` определяет, что у реплики есть `id`, `characterId`, `text` и опциональный массив `variants`.
- **`nodePositions`**: Важное поле в типе `Project`. Это словарь, где ключом является ID узла (сцены, списка диалогов или диалога), а значением — объект `{ x, y }` с его координатами в `FlowEditor`. Именно это поле обеспечивает сохранение расположения узлов.

### `utils/flowUtils.ts`

Этот файл содержит вспомогательные функции для работы с `ReactFlow`.

#### `buildNodesAndEdges(project, chapterId)`

- **Назначение**: Это ключевая функция-трансформер. Она берет "сырые" данные проекта (`project`) и ID активной главы (`chapterId`) и на их основе конструирует два массива: `nodes` и `edges`, которые понятны `ReactFlow`.
- **Логика**:
    1. Находит в проекте данные по активной главе.
    2. Итерируется по `scene`-ам этой главы и создает для каждой из них узел типа `scene`.
    3. Для каждой сцены итерируется по ее `dialogLists` и создает вложенные узлы типа `dialogList`.
    4. Для каждого списка диалогов итерируется по его `dialogs` и создает вложенные узлы типа `dialog`.
    5. **Создание связей**: Функция также создает связи. В текущей реализации она создает только вертикальные связи между последовательными диалогами внутри одного `DialogList`.
- **Важное замечание**: Как указано в TODO-комментарии в коде, функция в ее текущем виде **не создает** связи между сценами или между списками диалогов. Эта логика, вероятно, потребует добавления в типы данных полей вроде `nextTargetId`, чтобы явно указывать, куда ведет та или иная ветвь сюжета.

### `data/statIcons.ts`

Это простой файл данных, который не содержит логики.

- **Назначение**: Экспортирует большой объект `statIcons`, который используется компонентом `IconPicker` (внутри `StatInspector.tsx`) для отображения сгруппированного списка иконок для выбора.
- **Структура**: Объект имеет древовидную структуру:
    - Ключи верхнего уровня — это названия категорий (например, "Физические", "Интеллектуальные").
    - Каждая категория содержит два поля: `рекомендуемые` и `альтернативные`, которые, в свою очередь, содержат наборы иконок-эмодзи.
