Алгоритм построения и анализа графа истории (финальная версия с учётом статов, вариантов и ошибок)

1. Общая концепция

Граф истории включает:
1.	Сценарный уровень — главы и сцены.
2.	Диалоговый уровень — списки диалогов и переходы внутри сцены.
3.	Систему анализа — симуляция всех возможных прохождений с учётом статов и выбранных вариантов, выявление недостижимых узлов и ошибок.

Ключевые принципы:
•	Начальная точка истории:
Первая глава по order → её initialSceneId → initialDialogListId.
•	Статы (int) никогда не сбрасываются при переходах между главами, сценами или списками.
•	Переход из сцены возможен только после прохождения её конечного списка диалогов.
•	Циклы в графе разрешены, если из них можно выйти хотя бы одним валидным путём.

⸻

2. Типы узлов и термины

2.1. Глава (Chapter)
•	UUID, order, initialSceneId, название, описание.
•	Все сцены принадлежат одной главе.
•	Пустые главы в готовой истории — критическая ошибка.
•	Последняя существующая глава — финал истории.

2.2. Сцена (Scene Node)
•	UUID, название, описание.
•	initialDialogListId — ID начального списка диалогов (обязателен).
•	nextAvailableScenesUUID — ID следующих сцен.
•	Условия запуска — требования по статам и выбранным вариантам.
•	Списки диалогов.
•	Если хотя бы один список диалогов есть — он является конечным.

2.3. Список диалогов (Dialog List Node)
•	UUID, название, описание.
•	nextAvailableDialogListUUID — ID списков в этой же сцене (межсценовые переходы запрещены).
•	Условия запуска — требования по статам и выбранным вариантам.
•	Конечный список — nextAvailableDialogListUUID пуст.

2.4. Диалог (Dialog Frame Node)
•	UUID, текст/описание действия.
•	Варианты:
•	изменения статов (+/- int);
•	могут влиять на возможные пути в истории;
•	не создают прямых переходов (диалоги идут последовательно в рамках списка).

⸻

3. Построение графа

Этап 1 — Инициализация
•	Очистка состояния.
•	Проверка входных данных:
•	все initialSceneId существуют;
•	у каждой сцены есть initialDialogListId;
•	нет битых ссылок в nextAvailableScenesUUID и nextAvailableDialogListUUID — критическая ошибка;
•	order уникален, без пропусков и дублей.

Этап 2 — Внутрисценовые связи
•	Создаём рёбра между списками внутри сцены по nextAvailableDialogListUUID.

Этап 3 — Межсценовые связи
•	Для каждой сцены:
•	перебрать nextAvailableScenesUUID;
•	создать рёбра от всех конечных списков этой сцены → initialDialogListId целевой сцены.

Этап 4 — Межглавовые связи
•	Сортировать главы по order.
•	Для главы N:
•	конечные списки всех сцен → initialDialogListId сцены из initialSceneId главы N+1.
•	Пустая глава → критическая ошибка.

⸻

4. Анализ

4.1. Начальные условия
•	Статы — из настроек проекта, по умолчанию 0.
•	Множество chosenVariants пустое.
•	Статы не имеют фиксированных глобальных максимумов/минимумов, могут быть отрицательными.

4.2. Симуляция
•	При переходе:
1.	Проверка условий (сначала сцена, потом список).
2.	Применение эффектов выбора (+/- статы).
3.	Переход по ребру.
•	Если условия не выполнены — переход невозможен.

4.3. Ветвление
•	Каждый выбор создаёт отдельную ветку состояния (nodeId, stats[], chosenVariants[]).
•	ID вариантов уникальны (uuid), повторный выбор невозможен.
•	chosenVariants хранится как отсортированное множество для корректного сравнения состояний.

5. Расчёт пяти значений для каждого стата в узле

Для каждого узла u и стата s вычисляются:

5.1. Теоретические границы
•	min_possible(u,s):
•	-∞, если достижим цикл с w_s < 0, из которого есть путь в u.
•	Иначе — минимальная суммарная стоимость пути старт→u (учитывая условия).
•	max_possible(u,s):
•	+∞, если достижим цикл с w_s > 0, из которого есть путь в u.
•	Иначе — максимальная суммарная стоимость пути старт→u.

Техника:
•	Сжимаем граф в компоненты сильной связности (SCC).
•	Проверяем знаки весов циклов.
•	Для DAG после сжатия — находим мин/макс пути (Bellman–Ford или модифицированный DFS).

5.2. “Средние” по выборке достижимых состояний
1.	Запускаем симуляцию с ограничителями:
•	K — максимум проходов одной SCC в рамках пути (по умолчанию 1–2).
•	L — максимум длины пути (по умолчанию 1000).
•	Отсечение доминируемых состояний (Парето) по (nodeId, chosenVariants).
2.	Получаем множество значений X для стата s в узле u.
3.	Вычисляем:
•	mean — среднее по всей выборке.
•	lower_mean — среднее по нижним p% значений (p=25).
•	upper_mean — среднее по верхним p% значений (p=25).

5.3. Обработка бесконечностей
•	Если max_possible = +∞ → upper_mean и mean помечаются как diverges/undefined (можно дополнительно указать эмпирическое при K/L).
•	Если min_possible = -∞ → lower_mean помечается аналогично.

⸻

6. Ошибки

6.1. Критические
•	Сиротские узлы (сцены или списки, кроме стартовых).
•	Невозможные требования (нет ни одного пути, ведущего к узлу).
•	Отсутствие начального списка у сцены.
•	Пустые главы.
•	Битые ссылки (nextAvailableScenesUUID или nextAvailableDialogListUUID).
•	Тупиковые узлы, если это не финал истории.
•	Цикл, из которого нет выхода.

6.2. Предупреждения
•	Слишком короткий (<3 символов) текст диалога.
•	Слишком длинный (>300 символов) текст диалога.
•	Пробелы в начале/конце текста.
•	Множественные пробелы (\s{2,}).
•	Некорректные символы (по whitelist Unicode).

⸻

7. Логика обработки ошибок
   •	Все ошибки содержат:
   •	nodeType (chapter/scene/dialogList/dialog),
   •	nodeId,
   •	path (Chapter→Scene→DialogList→Dialog),
   •	errorCode,
   •	severity (critical или warning),
   •	message,
   •	meta (доп. данные).
   •	Critical → прерывает построение/анализ.
   •	Warning → фиксируется в отчёте, граф строится.

⸻

8. Производительность
   •	Построение графа: O(S + D + E).
   •	Анализ путей: экспоненциальный рост по числу ветвлений и размеру (stats, chosenVariants); оптимизации:
   •	дедупликация по ключу состояния;
   •	отсечение доминируемых состояний;
   •	ограничители K и L.

9. Конечный результат анализа

После завершения работы алгоритма система должна сформировать единый отчёт, содержащий:

9.1. Структурная карта истории
•	Полный граф всех узлов и рёбер:
•	главы → сцены → списки диалогов → диалоги;
•	ссылки между сценами и между главами;
•	внутренняя структура переходов внутри сцен.
•	Отметки типа узлов:
•	стартовые;
•	конечные;
•	финальные точки истории.
•	Информация о доступности каждого узла (достижим/недостижим).

9.2. Статистический профиль каждого узла

Для каждого узла и каждого стата:
•	min_possible — минимальное возможное значение стата.
•	lower_mean — среднее по нижним p% (дефолт 25%).
•	mean — среднее по всем достижениям.
•	upper_mean — среднее по верхним p% (дефолт 25%).
•	max_possible — максимальное возможное значение стата.
•	Пометка +∞/-∞, если граница теоретически бесконечна.

9.3. Путь достижимости
•	Количество уникальных способов достичь каждого узла.
•	Минимальная длина пути (в переходах) от старта.
•	Возможность попадания в узел при разных наборах выбранных вариантов.

9.4. Список ошибок

Критические
•	Все найденные критические ошибки (сироты, битые ссылки, невозможные требования, пустые главы, тупики, циклы без выхода и др.).
•	Для каждой ошибки:
•	nodeType, nodeId, полный path.
•	Код ошибки (errorCode).
•	Сообщение.
•	Доп. данные (meta), например:
•	требуемое значение стата vs достижимое;
•	ID битой ссылки.

Предупреждения
•	Замечания по качеству текста (короткий/длинный, пробелы, символы).
•	Аналогичная структура данных, но severity = warning.

9.5. Метаданные анализа
•	Дата и время запуска.
•	Параметры анализа (значения K, L, p).
•	Количество обработанных узлов, рёбер и состояний.
•	Время выполнения анализа.
