## Резюме

Добавляем в веб-конструктор **полноценный текстовый режим** редактирования активной главы — альтернативу графу сцен (React Flow). Автор пишет и правит сценарий в **расширенном DSL** с подсветкой структуры и валидацией; переключатель **Граф ↔ Текст** работает на одном проекте в store без потери данных. Главный риск — **двусторонняя синхронизация**: текущий `TextFormatService` неполный (экспорт теряет связи, требования статов, типы сцен и др.), а импорт через модалку перезаписывает проект целиком.

## Что делаем

Сейчас автор создаёт историю **визуально**: граф сцен и списков диалогов, правая панель-инспектор для реплик, вариантов и метаданных сцены. Текстовый путь существует только как **модалка импорта** (`ImportTextModal`) и внешний AI-пайплайн в `import/` + `docs/web/scenario-import/` — это не режим редактирования.

Нужен **peer-режим** «Текст»:
- полноэкранный редактор активной главы (сцены, списки диалогов, реплики, варианты, связи);
- **расширенный DSL** — покрывает всё, что сегодня задаётся в визуальном инспекторе (типы реплик, модификаторы, варианты с `(список N)`, требования по статам, тип сцены, аудио/фон как ссылки или описания — точный перечень на reconcile);
- **структура и валидация в редакторе**: ошибки синтаксиса, неизвестный персонаж, битая ссылка на список, дубли номеров — inline, до сохранения; структурный анализ графа (`AnalysisService`) — по запросу, как сейчас на графе;
- **переключатель** Граф ↔ Текст: одна правда в `storyStore`, смена режима без отдельного «импорта».

В store уже объявлен `viewMode: 'script'`, но UI не реализован — заложим его как точку входа.

## Для кого / зачем

**Автор истории** в веб-конструкторе Limerence, который:
- привык писать сценарий в Word/Google Docs и хочет править **линейно**, а не кликать по узлам;
- импортирует главы через `canonical.txt`, но сейчас вынужден снова уходить в граф для правок;
- хочет видеть **всю главу одним документом** и быстро менять реплики/ветки.

Визуальный граф остаётся для **отладки связей** и обзора структуры; текст — равноправная альтернатива, не «разовый импорт».

## Зафиксированные решения (из brief)

| Вопрос | Решение |
|--------|---------|
| Пользователь | Автор в веб-конструкторе |
| Объём текста | **Только активная глава** (сцены + диалоги); персонажи, статы, метаданные книги — вне текстового документа |
| Формат | **Расширенный DSL** на базе canonical (`import-text-guide`), но с полями инспектора |
| Связь с графом | **Переключатель** flow ↔ text, общий `Project` в store |
| v1 вне scope | AI-пайплайн в Cursor, мобильное приложение, монетизация в тексте, генерация арта/музыки |

## Acceptance criteria

- [ ] В toolbar редактора есть переключатель **«Граф» / «Текст»**; при смене режима активная глава отображается без потери несохранённых правок (с предупреждением, если есть конфликт).
- [ ] Режим «Текст» — **полноэкранный редактор** активной главы (не модалка): monospace, прокрутка, номера строк или навигация по заголовкам `###` / `##`.
- [ ] Текст главы **сериализуется из** и **применяется к** `Project` через единый слой (расширение `TextFormatService` или преемник); правки в тексте обновляют сцены/диалоги активной главы в store.
- [ ] **Расширенный DSL** документирован в `docs/web/` и покрывает минимум: главы/сцены/списки, типы реплик и модификаторы, `#Выбор` с `(список N)`, требования по статам на вариантах/переходах — перечень полей финализируется на reconcile по инспекторам.
- [ ] **Inline-валидация** в редакторе: синтаксис, нумерация списков, ссылки `(список N)`, имена персонажей из проекта; ошибки с номером строки и понятным текстом на русском.
- [ ] Сохранение (`S`) и загрузка с сервера работают **одинаково** из обоих режимов.
- [ ] Запуск **анализа структуры** (`R` / кнопка в дереве) доступен из текстового режима и показывает те же классы ошибок, что и на графе.
- [ ] Существующий **импорт из файла** (`ImportTextModal`) остаётся рабочим; поведение merge vs replace согласовано с текстовым режимом (не ломаем авторов pipeline `canonical.txt`).

## Вне scope

- Встроенный **AI-пайплайн** (`docs/web/scenario-import/`, Cursor) — остаётся внешним; конструктор только принимает готовый текст.
- **Мобильное** приложение и читательский runtime.
- **Монетизация**, настройки книги, обложка, жанры — только в `StorySettingsPage`, не в DSL главы.
- **Генерация** фона, спрайтов, BGM из текста (промты в инспекторе) — v1 не трогаем; в DSL допустимы только текстовые поля-описания, если reconcile решит включить.
- Редактирование **персонажей и статов** целиком в тексте — отдельные экраны (`CharacterEditor`, ProjectTree).
- **Совместное редактирование**, комментарии, история версий.
- Полный **round-trip всей истории** одним файлом — только активная глава.

## Глоссарий

| Термин | По-русски |
|--------|-----------|
| Конструктор / Builder | Веб-приложение автора: `web/editor-app/` |
| Visual flow / граф | Холст React Flow: узлы сцен и списков диалогов, рёбра переходов |
| Text mode / режим «Текст» | Полноэкранный редактор DSL активной главы |
| DSL / canonical format | Текстовый формат с заголовками `####`/`###`/`##` и строками `#Речь`, `#Выбор` |
| Глава (Chapter) | Часть истории; в текстовом режиме редактируется одна активная глава |
| Сцена (Scene) | Блок с фоном и цепочкой списков диалогов; в DSL — `### Сцена: …` |
| Список диалогов (Dialog list) | Группа реплик; в DSL — `## Список диалогов N` |
| Реплика / диалог (Dialog) | Строка `#Тип, Персонаж, текст` |
| Вариант (Variant) | Строка выбора с переходом `(список N)` |
| `TextFormatService` | Сервис parse/export текста ↔ `Project` в конструкторе |
| `AnalysisService` | Проверка связности и циклов графа после редактирования |
| `viewMode: 'script'` | Зарезервированное значение store для текстового режима (UI нет) |
| `canonical.txt` | Файл из AI-pipeline, готовый к импорту в конструктор |
